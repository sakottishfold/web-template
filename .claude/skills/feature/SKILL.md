# feature

## 最短原則

**単一ファイルではなく、コードベース全体を実装する心意気で**。ただし最短で ── 次をブロック解除する最小限を、確立パターンに従って書く。再利用 > 再作成。新しく書く前に必ず grep。

## 使うとき

- 機能 / ページ / route / コンポーネントの実装、バグ修正、リファクタ、スキーマ変更、新依存

## 使わないとき

- ドキュメントのみ → 該当 doc スキル / 探索のみ → スパイクと明示し本番出荷しない

## preflight(コードに触れる前に必須)

| #   | 参照                                       | 得られるもの                                   |
| --- | ------------------------------------------ | ---------------------------------------------- |
| 1   | `docs/spec/NNNN-*.md`(あれば)              | 何をすべきか、スキーマ、API 形状、受け入れ条件 |
| 2   | `docs/design/NNNN-*.md`(UI なら)           | 画面・状態・トークン・a11y                     |
| 3   | `docs/rules/code.md`(+UI なら `design.md`) | 検証ゲート・スタック規約・トークン             |
| 4   | `docs/adr/` 直近                           | 今のコードがそうなっている理由                 |
| 5   | 既存の類似コード                           | 真似るべきパターン(最も近い類例)               |

- **spec は条件付き**: 2 層以上 / 非自明な機能なら spec 必須(無ければ `spec` を先に呼ぶ)。単一ファイル / 単一レイヤーの軽微な変更は spec 無しでよく、`docs/PROJECT.md` + 依頼意図を契約に進める。
- 3 が欠ける、または「非自明なのに spec が無い」場合は止まって相談。コンテキスト無しの実装はパターンをフォークする。

## ワークフロー

1. **スコープ確認**: 単一 / 横断。サブタスクの集合(migration + API + UI + 統合)なら、**今回やらない分を `docs/NEXT-ACTIONS.md` に繰り延べ**(黙ったスキップ禁止)。
2. **既存コード監査**: 再利用できる util / hook / component / Server Action / 型を grep。`cn()`・shadcn・`src/lib/*` を優先。
3. **計画**: 横断ならファイルを触る順と最後の sweep を先に決める。
4. **実装**: 確立パターン準拠、トークン参照(ハードコード禁止)、契約(防御 vs 信頼)を明示、未確定 API には狭い props。
5. **検証ゲート(交渉不可)**: `pnpm check`(lint / format:check / typecheck / knip / test)が通ること。UI なら dev server を立て、変更ルートを `docs/PROJECT.md` のデザイン原則 / NOリストに照らしてビジュアル確認。
6. **consistency sweep**: 型変更→全呼び出し元、リネーム→import と doc、スキーマ変更→spec とmigration、env 追加→`.env.example`。
7. **doc 同期 + 静的化**: spec が実態と乖離→更新。新 export→発見可能性を担保。**今回の学びが lint / 型 / test で防げるなら、prose を書く前にそのガードを実装する**(`.oxlintrc.json` / test 追加など)。

## TDD(条件付き点火 ── 常時必須にはしない)

最短を守りつつ正しさを上げるため、**次のときだけ** `superpowers:test-driven-development` を借りて red→green で進める:

- **非自明なロジック**(分岐・計算・状態遷移・バリデーション)
- **バグ修正** ── まず**失敗する再現 test を書いて red にしてから**直す(回帰を二度と起こさない)
- 認証 / 認可 / データ整合性に関わる箇所

CRUD の薄いラッパ・設定変更・UI の見た目だけ、なら不要(検証ゲートで足りる)。判断は `docs/rules/quality.md`(機能正しさ)に従う。

## 並列実装(任意・横断機能で有効)

機能が独立レイヤー(DB / API / UI)に割れるなら、プロジェクトローカルの実装エージェントを**並列 dispatch**してよい(`.claude/agents/`):

- `db-implementer` ─ `src/db/**`・migration(spec の「スキーマ差分」を契約に)
- `api-implementer` ─ `src/server/**`・`AppType`(spec の「API 形状」を契約に)
- `ui-implementer` ─ `src/app/**`・`src/components/**`(design 仕様を契約に)

各 agent は所有ファイルが重ならないので競合しない。spec / design を**契約**として各レイヤーが独立に進み、最後に統合(狭い props → アダプタ)する。単一ファイルの小さな変更なら dispatch せず inline でよい。

## 完了前チェックリスト

- [ ] preflight 1〜5 を読んだ
- [ ] 既存を監査し再利用した(サイレントフォークなし)
- [ ] ハードコード色 / サイズなし ── 全てトークン
- [ ] `pnpm check` が通る
- [ ] UI: デザイン原則 / NOリスト / コントラスト AA を確認
- [ ] スキーマ / API 変更 → spec 更新、migration 生成
- [ ] 繰り延べたサブタスク → NEXT-ACTIONS に明示
- [ ] 契約(防御 vs 信頼)をコメント / 型で明示
- [ ] 機械化できる学び → lint / test に落とした(prose に逃げていない)

## 危険信号 ─ STOP

- 「小さい変更だから preflight 不要」「とりあえずハードコード」「動くから check 不要」「1ファイルだから sweep 不要」── すべて preflight 先・パターン次・検証は常に、のサイン。
