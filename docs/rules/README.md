# Rules

プロジェクト内のルール置き場。category別に分かれている。

- スキルは preflight で **該当 category だけ** を読む(全部は読まない)。
- `dev` が各ステップの学びをここに1行追記する。
- ルールが肥大/重複したら `dev` が compaction(統合・要約)する。

## 大原則: 静的ガードを優先(無作為に追記しない)

ここ(prose のルール)は**最後の手段**。学びを記録する前に必ず問う:

> 「これは lint / format / 型 / test / knip / CI で機械的に防げないか?」

- **防げる → そちらに実装する**(`.oxlintrc.json` にルール追加、型で縛る、test を書く、knip 設定、CI ゲート)。`docs/rules` には書かない。
- **防げない(判断・美意識・運用の機微)→ ここに1行追記する**。

理由: prose のルールは読まれない/腐る/肥大する。実行可能なガードは毎回自動で効く。`docs/rules` を「機械化できなかった残り」だけに保つことで、軽く保てる。

| ファイル      | 何のルールか                         | 主に読むスキル       |
| ------------- | ------------------------------------ | -------------------- |
| `code.md`     | 実装・コード規約・検証ゲート         | feature              |
| `design.md`   | デザイン・世界観・アクセシビリティ   | design, feature(UI)  |
| `docs.md`     | ドキュメント運用(ADR/spec/索引)      | spec, adr, dev       |
| `workflow.md` | 進め方・最短原則・チェックポイント   | 全スキル             |
| `quality.md`  | 品質マップ(特性→担当→機構・優先順位) | dev, harness-auditor |

## 追記フォーマット

```
- <ルール本文(命令形・1行)>  ── 由来: <ADR番号 or 機能id or 日付>
```

由来を残すと、後でルールの根拠を辿れる。
