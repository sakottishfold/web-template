# Workflow rules

進め方の共通鉄則。**全スキル**が読む。

## 最短で作り切る

- 次のステップを**ブロック解除する最小限**だけ書く/作る。gold-plating 禁止。
- 完璧な doc より、動く次の一歩。doc は後から学びで育てる。
- YAGNI ── 「将来使うかも」は作らない。必要になってから足す。

## チェックポイントは要所だけ

- 人間に確認するのは原則2点: **spec 確定時** と **実装着手前(design 後)**。
- それ以外は自律で進める。確認の取りすぎは最短に反する。

## サイレントを許さない

- 今回やらないサブタスクは `docs/NEXT-ACTIONS.md` に繰り延べる(黙ったスキップ = 後で忘れる)。
- パターンを外す/契約を選ぶ/依存を足すときは理由を残す。

## 学びを必ず回収(ただし静的ガード優先)

- 各ステップ後に「今回の学びは?」を問う(`dev` の責務)。
- **記録先の優先順位**(上から順に検討する):
  1. **機械的に強制できる** → lint(`.oxlintrc.json`)/ 型 / test / knip / CI ゲートに実装する
  2. 意思決定だった → `adr`
  3. doc がドリフトした → spec/design を更新
  4. スキル自体が不足/誤り → 該当 `SKILL.md` を更新
  5. **上のどれでもない判断・機微** → 初めて `docs/rules/<category>.md` に1行追記
- `docs/rules` に prose を足すのは最後の手段。無作為に追記しない。

## preflight を飛ばさない

- 作業前に該当 `docs/rules/*` と関連 doc(PRD/spec/design/直近ADR)を読む。
- コンテキスト無しに書くとパターンをフォークし、サイレントドリフトする。
